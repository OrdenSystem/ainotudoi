# 入所系3サービス追加 計画ドラフト（福祉型障害児入所施設／短期入所／日中一時支援）

- 日付: 2026-06-21
- ステータス: **ドラフト**
- 対象システム: 障害福祉事業所システム（AppSheet + Salesforce + GAS + Cloud SQL）

> ⚠️ **本計画はドラフトであり、Phase 0 の宿題2件（後述）が揃うまで実装着手しない。**
> Phase 0 が未完のまま実績項目設計・ひな型シート設計・GAS 分岐実装を始めると、リタリコ取込フォーマット不一致による手戻りが確定するため。

---

## 1. 目的・スコープ・非スコープ

### 1.1 目的
現行の相談支援系（計画相談支援・障害児相談支援）に閉じている本システムを、**入所系3サービス**まで拡張し、日次実績登録 → 月次集計 → リタリコ取込 Excel 連携の一連を同じ運用基盤で回せるようにする。

### 1.2 スコープ（本計画で扱う3サービス）
1. **福祉型障害児入所施設**（児童福祉法・入所）
2. **短期入所**（障害福祉サービス・ショートステイ）
3. **日中一時支援**（地域生活支援事業＝**市町村事業**）

各サービスについて: ServiceType 追加 → 受給者証・在籍状況のモデル整備 → AppSheet 実績登録 View → CloudSQL 日次レコード → GAS 月次集計＆リタリコ取込 Excel 生成。

### 1.3 非スコープ（本計画では扱わない）
- 国保連 CSV の直接生成（現行どおりリタリコ取込に委ねる方針を踏襲）。
- 生活介護・施設入所支援など既存 ServiceType の請求実装（ServiceType 値は存在するが本件対象外）。
- 報酬改定・単価マスタの自動追従（単価は手運用 or 別マスタ。本件は「現行の単価管理方法に乗せる」前提）。
- AI 機能（録音・記録生成）の新サービス向けプロンプト最適化（既存資産を流用し、調整は別計画）。
- Salesforce 側の請求ロジック（請求は GAS + リタリコのまま）。

## 2. 現状アーキテクチャ要約と再利用できる資産

### 2.1 現状フロー（確認済み）
```
SFマスタ（Customer/Office/DisabilityCard/CustomerStatus）
  → AppSheet が事業所/利用者情報を取得
  → AppSheet で日次実績登録・記録・AI機能
  → CloudSQL「01相談記録」（年月/日/基本報酬/加算 の日次レコード）
  → GAS（001_レセプト生成）が category 文字列で分岐 → 月次集計
  → ひな型スプシ makeCopy → category シート選択 → ラベル→値マッピング
  → 「日報Excel置換」シートでリタリコ取込フォーマットへ最終調整
  → リタリコ取込（国保連CSVは作らない）
  → CloudSQL「フラグ」を請求済(TRUE)に更新（前後CSVバックアップあり）
```

## 3. ギャップ分析

| 項目 | 現状 | 福祉型障害児入所 | 短期入所 | 日中一時支援 |
|---|---|---|---|---|
| **ServiceType__c 値** | 短期入所=あり / 生活介護・施設入所=あり | **無し→追加** | あり（流用） | **無し→追加** |
| **支給量（日数）項目** | DisabilityCard に**無い** | 必要（入所日数の上限管理） | 必要（短期利用日数の上限） | 市町村事業のため日数概念が別 |
| **上限管理事業所項目** | DisabilityCard に**無い**（上限管理有無・上限額のみ） | 必要（多事業所利用時の上限管理者特定） | 必要 | 地域生活支援＝上限管理の扱いが別 |
| **契約オブジェクト** | **無し** | 措置/契約区分の管理が必要 | 契約日・契約支給量 | 市町村との関係（契約というより支給決定） |
| **GAS category 分岐** | "計画相談支援"/"障害児相談支援" の2分岐 | category 追加＋集計ロジック | category 追加＋集計ロジック | **別系統**（市町村別単価ロジック） |
| **ひな型スプシ・シート** | 相談2サービス分のシートのみ | 新シート（要テンプレ） | 新シート（要テンプレ） | 独立テンプレ（市町村別の可能性） |
| **日報Excel置換シート** | 相談用のみ | リタリコ取込仕様に合わせ新規 | 同左 | **独立仕様の可能性大** |

### 3.1 ギャップの本質
- **データモデル不足**: 支給量（日数）・上限管理事業所・契約区分 を保持する場所が SF に無い。3サービスとも日数ベース請求のため日数上限が必須。
- **GAS 分岐とテンプレ依存**: 集計エンジンは流用可だが、出力テンプレ（category シート＋日報Excel置換）が「リタリコ取込仕様の実体」であり、これが無いと完成しない。→ **Phase 0 のブロッカー**。
- **日中一時支援の別系統性**: 市町村事業のため請求系統・テンプレ・単価が他2つと独立（後述の独立節）。

## 4. Phase 0（着手の前提条件・ブロッカー）

> **このフェーズが完了するまで Phase 2 以降の実装に着手しない。**

### 宿題(1): リタリコの3サービス分の取込 Excel テンプレ取得【ブロッカー】
- **内容**: 福祉型障害児入所施設 / 短期入所 / 日中一時支援 それぞれのリタリコ取込 Excel テンプレを入手（**空でよい。列名・シート名・必須列がわかるもの**）。
- **理由**: リタリコ取込仕様の実体はコードでなくテンプレ＋置換シート側にある。これが無いと「実績で取るべき項目」も「ひな型シート」も「日報Excel置換」も設計できない。推測で項目を決めると取込時の列不一致で全面手戻りになる。
- **成果物**: 3テンプレファイル（or スクショ）＋ 列名・シート名一覧。日中一時は市町村別に複数様式があるか確認。
- **受け入れ条件**: 各サービスのテンプレについて「シート名／列ヘッダ／日数・加算が入る列」がドキュメント化されている。

### 宿題(2): 既存ひな型スプシ／「日報Excel置換」シートの場所(URL)特定【ブロッカー】
- **内容**: 現行の相談用ひな型スプレッドシートと「日報Excel置換」シートの **URL（Drive 上の位置）** を特定。
- **理由**: 新サービス用シートを既存に倣って設計するため。makeCopy → category シート選択 → ラベル→値マッピング の実体を読まないと、新 category シートの構造を既存準拠で作れない。
- **成果物**: ひな型スプシ URL ＋ 「日報Excel置換」シート URL（同一ブック内か別ブックか含む）。GAS のどの定数/プロパティに ID がハードコードされているかも併記。
- **受け入れ条件**: planner/implementer が当該スプシを開いて category シートの命名規則・マッピング行構造を確認できる状態。

> Phase 0 のいずれかが**スプシ/GAS の Drive 位置探索**を伴う場合は `google-workspace-ops` スキルを参照。Cloud SQL のスキーマ拡張設計に入る前に `postgres-patterns`（PostgreSQL 前提）を参照する。

## 6. サービス別 実績項目設計の論点

### 6.1 福祉型障害児入所施設
- **基本**: 入所日数（在籍日数ベース）、入退所日。
- **控除**: 外泊控除、入院控除（日数からの減算ロジックが集計に必要）。
- **区分**: 措置／契約 の別（請求先・様式が変わる）。
- **児童特有**: 保護者情報・児童相談所（児相）の関与。要配慮個人情報として扱い厳格に。
- **加算**: 各種加算（要テンプレで列確認）。
- **請求先**: 国保連（児童）。リタリコ取込テンプレ＝児童様式。

### 6.2 短期入所
- **基本**: 利用日数（実日数）。
- **付随**: 送迎、食事提供。
- **類型**: 単独型／併設型／空床利用型（単価が変わるため実績で類型を保持）。
- **加算**: 緊急短期入所受入加算 等。
- **請求先**: 国保連（介護給付）。リタリコ取込テンプレ＝介護給付様式。

### 6.3 共通の集計論点
- 日数ベース請求のため「日次レコードを月次で日数集計」する既存エンジンがそのまま効く。福祉型児童入所・短期入所は **category 追加で乗る見込み**。
- ただし外泊/入院控除・利用類型は単純な日数カウントでないため、控除・類型を反映する集計拡張が要る。

### 6.4 【独立節】日中一時支援は別系統（最重要注意点）
- **制度区分**: 地域生活支援事業＝**市町村事業**。障害福祉サービス（国保連）とは請求系統が根本的に異なる。
- **単価**: **市町村ごと・時間区分ごと**の単価。市町村別単価マスタが必須（保持先を §7 で決定）。
- **請求先**: 各市町村（国保連を経由しないケースが一般的）。
- **テンプレ**: リタリコ取込仕様が他2サービスと独立の可能性が高い。**市町村別に様式が分かれる可能性**もあり、Phase 0 で要確認。
- **設計方針**: 既存 category 分岐の延長で無理に統合せず、**日中一時専用の集計・出力ルート**を分けて設計する。共通化は集計エンジンの「日次→月次」部分に限定し、単価適用・出力テンプレは独立実装とする。
- **在籍/受給者証**: DisabilityCard.Type__c の「地域生活支援」区分を使う想定（要確認）。支給量も「日数」でなく市町村の支給決定（時間/回数）で表現される可能性。

## 9. 代替案（却下）

### 代替案A: 「01相談記録」を使わず新サービス専用の請求基盤を別途構築
- **却下理由**: 日次→月次集計エンジン・請求完了処理（フラグ＋CSVバックアップ）という再利用可能な核心資産を捨てることになり、開発量・保守コストが倍増する。少なくとも福祉型児童入所・短期入所は既存 category 分岐の延長で乗るため、共通基盤流用が合理的。日中一時のみ別ルートに切るのが最小コスト。

### 代替案B: リタリコ取込をやめて国保連 CSV を直接生成する
- **却下理由**: 非スコープに反する。現行が「国保連CSVは作らずリタリコに取込ませる」方針で安定稼働しており、リタリコ取込仕様（テンプレ＋置換シート）という資産がある。CSV 直生成は国保連伝送仕様の完全実装が必要で、本件の追加3サービスにそのリスクを負わせる必然性がない。

### 代替案C: Phase 0 を省略し、制度知識からテンプレ項目を推測して並走実装
- **却下理由**: リタリコ取込仕様の実体はコードに無くテンプレ／置換シート側にある。推測で列・シート名を決めると取込時の不一致で Phase 5 全面手戻り。Phase 0 のブロッカー化は手戻りコスト回避が目的。

### ロールバック観点
- **SF 項目追加**: 新規項目は既存に non-breaking（追加のみ）。問題時は View から外し、項目自体は残置（削除は影響大のため避ける）。
- **CloudSQL**: 新テーブル新設なら drop で巻き戻し容易。既存「01相談記録」拡張の場合は **追加カラムは nullable** にして既存相談請求を壊さない。
- **GAS**: category 分岐は **既存2 category の挙動を変えない**こと（追加分岐のみ）。clasp push 前にローカル検証、本番反映はユーザ承認後（過去の運用慣行に準拠）。
- **リタリコ連携**: 新サービス Excel 生成は既存生成と独立関数 or 独立 category シートに閉じ、相談請求の生成経路に副作用を出さない。
- **請求完了フラグ**: #9 でサービス別運用にする場合、誤フラグ時の復旧手順（前後CSVバックアップからの復元）を Phase 6 で確認。
