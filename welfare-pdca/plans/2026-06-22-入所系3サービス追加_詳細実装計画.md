## 第 1 部 エグゼクティブサマリ

### 1.1 目的

現在「計画相談支援」「障害児相談支援」の 2 サービスで稼働している障害福祉システム ainotudoi を、**入所系 3 サービス（児童入所施設・短期入所・日中一時支援）** に拡張する。同時に **令和 8 年 6 月の報酬改定**（計画相談・障害児相談への「福祉・介護職員等処遇改善加算」追加）に対応する。

両方の対応を 1 つの計画にまとめる根拠：

- 6 月改定への対応と入所系 3 サービス追加の運用開始タイミングが **2026-06 実績分**で重なる
- どちらも「ひな型スプシ新設」「GAS の月次条件分岐追加」「リタリコ取込 Excel テンプレ適用」という同じ作業軸上で進む
- どちらも既存稼働ロジックを **触らずに分岐を増やす**形で実装する点が同じ

### 1.2 スコープ

#### In scope

- Salesforce: `Office__c.ServiceType__c` ピックリストに 2 値追加、`DisabilityCard__c` に 9 項目追加、業務オブジェクト 3 個新設（暫定: `ChildCareEntryRecord__c` / `ShortStayRecord__c` / `DaytimeTempSupportRecord__c`）
- CloudSQL: 業務テーブル 3 個新設（`child_care_entry_records` / `short_stay_records` / `daytime_temp_support_records`）+ 補助テーブル 2 個（`municipalities` / `municipality_unit_prices`）
- AppSheet メインアプリ: 業務テーブル 3 個追加、View 約 15 個追加、Action 約 30〜60 個追加、業種種別マスタに 2 値追加
- AppSheet 請求アプリ: 既存 6 シート（事業所加算項目DB / 市町村情報DB / 利用者情報基本項目DB / 上限額管理状況DB / 日報Excel置換 / 請求情報DB）に新サービス用エントリ追加
- GAS: `executeMakeRecept()` の category 分岐に 3 件追加 / 月次条件分岐（6 月改定）追加 / 暦日生成 + LEFT JOIN ロジック追加
- ひな型スプシ: 入所系専用ブック新設（3 シート構成）+ 6 月改定対応スプシ新設（計画相談 + 障害児相談 2 シート）

#### Out of scope

- 国保連 CSV の直接生成（リタリコ取込を経由する現行運用を維持）
- 生活介護・施設入所支援など既存 ServiceType の請求実装（ServiceType 値は picklist に存在するが本件対象外）
- AI 機能（録音・記録生成）の新サービス向けプロンプト最適化（既存資産を流用、調整は別計画）
- Salesforce 側の請求ロジック実装（請求は GAS + リタリコ取込のまま）
- リタリコ単価マスタの自動追従（単価は手運用 or 別マスタ。本件は「現行の単価管理方法に乗せる」前提）
- 国保連電子請求の e-Tax 連携

### 2.7 ⚠️ 4 警告 — 既存 GAS 請求 Excel 出力ロジックの厳守事項

> ユーザー言明: 「請求 Excel 出力の GAS は以前すごく苦労した。あらゆる所に気を配って仕上げている。既存の GAS 処理を深く読むように計画に入れる」

以下の 4 点は **既存の挙動を完全に保持**すること。新サービス追加は「既存ロジックに 3 分岐追加」の形で行い、**既存ロジックの再設計はしない**：

#### 警告 1（事業所情報セクション）

- 事業所全体で取る加算項目は **本体システム（HopeCareDX）とは別の請求アプリ**（`請求_HopeCareDX_愛の集い` / App ID `f6ddf60e-a346-4d4c-a143-eeb9aed81287`）から取得している
- → 入所系 3 サービス追加でも、**事業所単位の加算項目は請求アプリで管理**する
- → AppSheet 設計時に「請求アプリのどのテーブルにどう紐付くか」を spec 化必須（第 6 部 §6.2 で詳述）

#### 警告 2（市町村情報の動的行）

- 市町村が複数ある事業所では市町村ブロックが繰り返され、それに応じて利用者ブロックの開始行がズレる
- **行番号をハードコードしている GAS 関数は、この動的計算ロジックを必ず通している**
- → 入所系追加でも **動的行番号計算ロジックを流用**、決して固定オフセットで決め打ちしない
- → 特に **日中一時支援（市町村事業）** は影響が大きい

#### 警告 3（テキスト置換処理の特殊性）

- 「日報Excel置換」シートで行うラベル→値の置換ロジックには **既存の特殊処理**がある
- → 既存の置換ルールに **新行・新ラベルのエントリを追加する形**で対応、既存エントリは変更しない

#### 警告 4（曜日セル数式保持）

- 利用者欄の曜日セルは **数式で自動生成**されている（サービス提供年月入力で 1〜31 日の曜日が自動反映）
- GAS が値を書き込むと数式が壊れる → **書込対象範囲を厳密に限定**している
- → 入所系追加でも **既存の書込範囲制御ロジックを継承**、新ロジックで範囲を再定義しない

### 2.8 ⚠️ §7 暦日固定行 — Excel 出力行構造の根本的な違い（既存 vs 新規）

> ユーザー追記事項（2026-06-22）

| 観点 | 既存（計画相談・障害児相談） | 新規（入所系 3 サービス） |
|---|---|---|
| 利用者ブロック内の日次行 | **実績がある日数分だけ行追加** | **暦日 1〜31 日（または月末日まで）すべての行が固定で並ぶ** |
| 空欄の意味 | 行自体が存在しない | 行は存在し、列の値が空 = 実績なし |
| 1 利用者ブロックの行数 | 可変（利用日数依存） | **固定（28〜31 行 + ヘッダ・属性領域）** |
| 月またぎ・空欄日の扱い | 行が無いだけ | 行はあり、データ部空 |

#### CloudSQL 設計への影響

- **CloudSQL は疎データのまま**（実績がある日のみ INSERT）。これは確定。
- GAS が Excel 出力時に「**暦日マスタ（1〜月末日）と LEFT JOIN**」して、空行も含めて 28〜31 行を出力する。
- Excel 出力ロジックの **暦日生成 + LEFT JOIN ステップ**を新サービス分で追加必要。

#### GAS の動的行番号計算への影響

既存 vs 新規で「利用者ブロック開始行 → 次の利用者ブロック開始行」の増分計算式が違う：

| 観点 | 既存 | 新規（入所系） |
|---|---|---|
| 増分式 | `currentBase + 属性行数 + 実績件数 + 余白` | `currentBase + 属性行数 + 暦日固定数(28〜31) + 余白` |
| ハードコード可否 | NG（実績件数で動的） | NG（月の日数で 28/29/30/31 と変動） |

→ category 別 + サービス提供月別の **暦日数取得関数**を GAS に追加する必要あり（第 5 部 §5.3）。

#### 「市町村情報の動的行」との合成（警告 2 と本節）

入所系の利用者ブロック開始行 = `事業所セクション末尾行 + 市町村ブロック数 × 5 + (前利用者ブロックの累計行数)`

このうち **「前利用者ブロックの累計行数」が暦日数 × N 人**で動くため、**市町村数と暦日数の両方を動的計算**する必要がある。**ハードコード絶対禁止**。

### 5.3 暦日生成 + LEFT JOIN ロジック（新サービス専用）

#### 5.3.1 暦日生成関数

`decisions §7` で確定した「暦日 1〜月末日すべての行が固定で並ぶ」要件に対応するため、以下のヘルパを新規追加：

```javascript
// 新規ファイル: 001_calendar_helper.js
/**
 * 指定された年月の暦日配列を返す（うるう年対応）
 * @param {number} year - 年（例: 2026）
 * @param {number} month - 月（1〜12）
 * @return {number[]} - [1, 2, 3, ..., 末日]
 */
function getDaysInMonth(year, month) {
  // JavaScript の Date.getDate() で月末日を取得（month は 0-indexed なので +1）
  const lastDay = new Date(year, month, 0).getDate();
  const days = [];
  for (let d = 1; d <= lastDay; d++) {
    days.push(d);
  }
  return days;
}

/**
 * YYYYMM 文字列を年月配列に分解
 * @param {string} yyyymm - 例: "202606"
 * @return {{ year: number, month: number, lastDay: number }}
 */
function parseYyyymm(yyyymm) {
  const year = parseInt(yyyymm.substr(0, 4), 10);
  const month = parseInt(yyyymm.substr(4, 2), 10);
  const lastDay = new Date(year, month, 0).getDate();
  return { year, month, lastDay };
}
```

|---|---|---|

| (2024, 2) | 29 | 29（うるう年） |
| (2026, 4) | 30 | 30 |
| (2026, 6) | 30 | 30 |

#### 5.3.2 CloudSQL 疎データ × 暦日マスタの LEFT JOIN ステップ

実装方針として、以下のいずれかを採用：

##### 方針 A：JavaScript 側で LEFT JOIN（推奨）

```javascript
// 新規関数: 001_dense_data_builder.js
/**
 * 疎データを暦日固定行に展開する
 * @param {Array<Object>} sparseRows - CloudSQL から取得した実績レコード（疎）
 * @param {string} yyyymm - "202606"
 * @return {Array<Object>} - 1〜末日すべての行（実績なし日は空オブジェクト）
 */
function buildDenseDailyRows(sparseRows, yyyymm) {
  const { lastDay } = parseYyyymm(yyyymm);

  // 「日」をキーにした Map を作成
  const sparseByDay = {};
  sparseRows.forEach((row) => {
    const day = parseInt(row.service_day || row["日"], 10);
    if (!isNaN(day)) sparseByDay[day] = row;
  });

  // 暦日配列を生成して LEFT JOIN
  const denseRows = [];
  for (let d = 1; d <= lastDay; d++) {
    if (sparseByDay[d]) {
      denseRows.push({ day: d, ...sparseByDay[d] });
    } else {
      // 実績なし日：day だけ持つ空行
      denseRows.push({ day: d });
    }
  }
  return denseRows;
}
```

##### 方針 B：PostgreSQL の generate_series で LEFT JOIN（参考）

`03-cloudsql-and-docs.md` §9 で示された SQL を参考に、SQL 側で暦日を生成して JOIN する方法もあるが、**GAS 側で完結する方針 A を採用**（GAS の既存パターンと整合）。

## 第 9 部 ロールバック手順

### 9.1 ロールバックの基本方針

各 Phase の変更は **逆順で巻き戻し可能**になるよう設計する。特に GAS / CloudSQL の変更は破壊的なので慎重に。

### 9.2 Phase 別ロールバック手順

#### 9.2.1 Phase 5（GAS 集計）のロールバック

| 変更 | ロールバック手順 |
|---|---|
| `executeMakeRecept` への分岐追加 | Git で改修コミットを revert → clasp push で本番反映 |
| Script Property 追加 | PropertiesService から `SS_SOURCE_URL_*` を削除 |
| 新規ヘルパファイル（`001_calendar_helper.js` 等） | Git で削除 → clasp push |
| 6 月改定の月次条件分岐 | revert で旧 SSsourceURL 受け取りロジックに戻す |

**注意**: GAS の clasp push は **本番反映の即時性が高い**。事前にユーザー承認を取り、Slack 通知して反映後 5 分は監視する。

#### 9.2.2 Phase 4（AppSheet）のロールバック

| 変更 | ロールバック手順 |
|---|---|
| 新規 3 テーブル追加 | AppSheet Editor → Data → 該当テーブル → Delete |
| 新規 15 View 追加 | UX → Views → 該当 View → Delete |
| 新規 Action 追加 | Behavior → Actions → 該当 Action → Delete |
| 業種種別マスタ 2 値追加 | テーブルの該当行を削除 |

**注意**: AppSheet の編集履歴は Editor の Version 機能で巻き戻し可能。Editor で Version Snapshot を取得しておく。

#### 9.2.3 Phase 3（CloudSQL DDL）のロールバック

```sql
-- 逆順で DROP（FK 制約があるため municipality_unit_prices → municipalities の順）
DROP TABLE IF EXISTS public.municipality_unit_prices;
DROP TABLE IF EXISTS public.municipalities;
DROP TABLE IF EXISTS public.daytime_temp_support_records;
DROP TABLE IF EXISTS public.short_stay_records;
DROP TABLE IF EXISTS public.child_care_entry_records;
```

**注意**:
- DROP 前に **データのバックアップ**を pg_dump で取得
- 既存「01相談記録」等の 7 テーブルへの影響はゼロ（独立テーブルのため）

#### 9.2.4 Phase 2（Salesforce）のロールバック

| 変更 | ロールバック手順 |
|---|---|
| ServiceType__c 値 2 件追加 | Setup → picklist の該当値を Inactive 化（**削除はしない**、レコード参照保持のため） |

**注意**:
- SF カスタム項目の削除は **慎重に**。既存レコードからの参照が残るとエラーになる
- 推奨は「Inactive 化 + 項目非表示」のソフトロールバック

### 9.3 緊急時のフォールバック

#### 9.3.1 リタリコ取込でエラーが発生した場合

1. 生成 Excel のシート構造・列名・行内容をリタリコのエラーログと突合

3. AppSheet 側の Editor Version を直前のものに戻す
