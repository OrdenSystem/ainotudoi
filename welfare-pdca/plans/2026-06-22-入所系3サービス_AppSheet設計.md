## 5. Action 設計

### 5.1 共通 Action テンプレート（3 テーブルそれぞれに同名で作成）

#### Action 1: 「請求完了」
```yaml
Name: <TableName>_請求完了
SourceName: <TableName>
ActionType: Data: set the values of some columns in this row
ColumnsToUpdate:
  flag: TRUE
  flag_nichiji: NOW()
Prominence: Display prominently
NeedsConfirmation: TRUE
ConfirmationMessage: 選択した行を「請求完了」に変更します。よろしいですか？
OnlyIfThisIsTrue: =NOT([flag])    # 未請求行のみ表示
SelectFromAvailableRows: TRUE     # 一括選択可
```

#### Action 2: 「再請求」
```yaml
Name: <TableName>_再請求
SourceName: <TableName>
ActionType: Data: set the values of some columns in this row
ColumnsToUpdate:
  saisei_flag: TRUE
Prominence: Display overlay
NeedsConfirmation: TRUE
ConfirmationMessage: 「再請求」フラグを立てます。再請求対象として扱われます。
OnlyIfThisIsTrue: =[flag]         # 請求完了済のみ表示
SelectFromAvailableRows: TRUE
```

#### Action 3: 「暦日一括生成」
最も複雑。**Form View** + **Composite Action** の組み合わせで実装。

```yaml
# 入口: ボタンクリック → 暦日Form を開く
Name: <TableName>_暦日一括生成_開く
ActionType: App: go to another view within this app
Target: LINKTOFORM(
  "<TableName>_暦日Form",
  "riyousha_zaiseki_id", _PARAM_or_CONTEXT,
  "nengetsu", TEXT(TODAY(), "YYYY-MM")
)

# Form 確定後の Bot (Process)
Bot: <TableName>_暦日一括生成_実行
Trigger: Form Saved (on 暦日Form)
Process:
  - For each day in EOMONTH(...) range:
      Add row to <TableName> with:
        record_id: UNIQUEID()
        riyousha_zaiseki_id: [入力値]
        kiroku_bi: 該当日
        nichi: TEXT(該当日, "DD")
        nengetsu: TEXT(該当日, "YYYY-MM")
        flag: FALSE
        toroku_nichiji: NOW()
```

> **設計上の注意**: AppSheet の Process（Bot）でループ生成は可能だが、CloudSQL の負荷を考慮し**サーバー側（GAS or stored procedure）でバッチ生成**する方が筋が良いケースも。AppSheet Process は数十行ならスムーズに動く。

#### Action 設計の選択肢

| 方式 | Pros | Cons |
|---|---|---|
| **AppSheet Process（Bot）** で Add Row ループ | UI 完結、リアルタイム | CloudSQL に N 回 INSERT で負荷 |
| **GAS API** を Webhook で呼出 | 単一トランザクションで高速 | GAS デプロイ・URL 管理が必要 |
| **PostgreSQL stored procedure** | DB 内で最速・integrity 高 | DDL 追加要、ロールバック大変 |

→ **Phase 1: AppSheet Process** で実装。負荷問題が出たら GAS に切替。

## 6. REF 関係（既存テーブルとの紐付け）

```text
入所系3サービス各テーブル
  ├─ riyousha_zaiseki_id  →  REF → CustomerStatus__c
  ├─ shokuin_zaiseki_id   →  REF → StaffStatus__c
  ├─ riyousha_id          →  REF → Customer__c
  └─ shichoson_id (日中のみ) → REF → 市町村マスタ (municipalities)

CustomerStatus__c
  └─ Related <TableName>s   ← 入所系3テーブルから REF_ROWS で逆参照
```

利用者詳細画面に「最近の児童入所記録」「最近の短期入所記録」を inline 表示できるよう、CustomerStatus__c の Detail View に `Related <TableName>s` 列を追加（フェーズ 2）。

## 7. ユーザー視点の業務フロー

### 7.1 日常入力（個別）

1. ホーム → 「児童入所記録」タップ
2. List View が開く → 「+」ボタンで Form 開く
3. 利用者選択 → 日付 → 状態 → 保存

### 7.2 月初一括生成 → 個別編集

1. ホーム → 「児童入所記録」→ メニュー「暦日一括生成」
2. Form: 利用者選択 + 年月選択 → 保存
3. 自動生成 28〜31 行が List に並ぶ
4. 各行を編集して当日の実情を入力

### 7.3 月末請求処理

1. List で対象月の行を絞り込み（手動 or 検索）
2. 全行選択 → 「請求完了」ボタン
3. 確認ダイアログ → 完了

## 9. 再利用可能スクリプトの設計

CloudSQL 適用後の AppSheet 実装は **saveapp 直接呼出しで自動化** することで、再現性とロールバック可能性を確保。

### 必要なスクリプト（Phase D-2 以降で追加予定）

```text
scripts/
  appsheet-add-dataset.py           # 既存 SQL 接続から新テーブル DataSet 追加

  appsheet-add-action.py            # Action（set column / link to form）追加
  appsheet-promote-to-menu.py       # Position: ref → menu に昇格
```

各スクリプトは既存 `appsheet-update-view-columns.py` と同パターン:
- loadApp → modify (in-memory) → saveapp
- 設定 JSON で複数操作を一括化
- dry-run / --apply 切替

### 10.1 個人設定への列追加方法
- 既存「個人設定」スプシは `1M75wbI08EbZ0OLtM7RFmsCj9Y1m66EV6hYmRfynNXYc`（_Schema は別ベース）
- カラム追加 → AppSheet 再認識のフローが必要
- → **ユーザーが Editor で実施**するのが簡単（Phase D-1 で 5 分作業）

### 10.2 単価マスタの初期データ投入
- `municipality_unit_prices` テーブル新設後、初期データ（市町村別単価）を投入する必要
- ソース: 各市町村の障害福祉サービス担当窓口に問い合わせ or 過年度実績 CSV
- → 別フェーズで対応（DDL 適用 ≠ データ投入）

### 10.3 暦日一括生成の重複防止
- 既に同月の暦日が生成されている時に再実行すると重複行が増える
- → Process 冒頭で `SELECT COUNT(*) WHERE riyousha_zaiseki_id = ? AND nengetsu = ?` を確認、>0 なら中止
- → または UNIQUE 制約（DDL 既存）が INSERT エラーで防ぐ（再請求行を除く）

### 10.4 入所系 3 サービス専用 受給者証 View
- 既存「障害福祉サービス受給者証」「障害児通所受給者証」と並ぶ 3 サービス向け Slice
- 例: `児童入所施設_受給者証` Slice (filter: `Office__r.ServiceType__c = "児童入所施設"` & `Valid__c`)
- → 必要かは運用観察してから（Phase 2 以降）
