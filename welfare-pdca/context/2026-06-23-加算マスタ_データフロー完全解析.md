## §1. 4 つの重大な発見

### 発見 1: 加算マスタは両アプリ共有

```
HopeCareマスタ AppSheet Database (base 0xwgjdrXPv4BIyuQb3buAd)
  ├── 001_事業所加算マスタ ─┐
  ├── 001_利用者加算マスタ ─┤── 主アプリ・請求アプリ
  └── 001_利用者基本マスタ ─┘    両方が UpdateMode=0 で読取専用参照
                                  使用事業所 EnumList で愛の集い分のみ表示
```

[decisions §6-1] の「事業所加算は請求アプリから取得」は **半分誤解**:
- **マスタ**: 両アプリ共有（HopeCareマスタ DB）
- **月次実績データ**: 請求アプリ専有（スプシ `事業所加算項目DB`）

### 発見 2: 主アプリには Bot / Workflow / GAS 呼出が無い

主アプリ HopeCareDX:
- `Behavior.AppBots = []`
- `Behavior.AppProcesses = []`
- `WorkflowRules = []`
- CallScript Action = 0 件

→ 加算は AppSheet Form 入力 → CloudSQL `01相談記録` 直書込のみ

### 発見 3: 請求アプリには Bot 4 本 + GAS 2 本が集約

```
請求情報DB (請求アプリ スプシ)
   ├── レセBOT (自動フラグ変化トリガ) → makeRecept GAS
   ├── 新規追加レコードbot → AddRecordCopyYYYYMM GAS
   ├── 請求完了bot (完了日時変化) → completeBillingWithIDs_CloudSQL GAS
   └── レセBOT 2 (File添付トリガ) → AddKasanCoding GAS

GAS scriptIds:
   11DEem...AI5Evvb → makeRecept + completeBillingWithIDs_CloudSQL
   1gIg-tZ...uLBA   → AddRecordCopyYYYYMM + AddKasanCoding
```

### 発見 4: 「反映条件」は GAS 未実装

GAS `001_レセプト生成_CloudSQL.gs` で `Grep "反映条件"` = **0 件**

→ 「同月最新日のみ反映 / 同月全件反映_カウント」は **AppSheet Bot 側で 01相談記録 行に展開する前段でのみ評価** される設計（または未実装）

現在の利用者加算マスタ 49 行 **全て** が「同月最新日のみ反映」。「同月全件反映_カウント」は **Enum 定義のみ存在、データ 0 行**。

## §3. 加算マスタの正しい使い分け

### 001_事業所加算マスタ（事業所単位・月次フラグ）
- **目的**: 体制届で固定された事業所全体の加算（月初〜月末で固定値）
- **既存例**: 相談支援機能強化型体制 / 主任相談支援専門員配置加算 / 情報公表未報告減算 / 業務継続計画未策定減算 / 実費1〜2 / 請求書役職・氏名
- **GAS が読む**: **読まない**（GAS は appDB の `事業所加算項目DB`（請求アプリ スプシの月次実績）を読む）
- **AppSheet UI**: `有効_事業所加算` Slice は存在するが、現状未参照（デッドコード）

### 001_利用者加算マスタ（利用者単位・日次/月次）
- **目的**: 利用者ごとに発生する加算
- **49 行すべて「同月最新日のみ反映」**（日数累積概念なし）
- **AppSheet UI**: 01相談記録 Form の「加算」「基本報酬」「区分選択肢」列の Valid_If で参照
- **GAS が読む**: **読まない**（既に 01相談記録 に文字列保存されているのを読む）

### 001_利用者基本マスタ（利用者属性ラベル）
- **目的**: 帳票ラベル定義（氏名、受給者証番号、支給市町村、児童氏名、児童氏名カナ、利用者負担上限額）
- **GAS が使う**: テンプレ A 列ラベル検索でこの列名と一致したら、B 列に該当値を書込
- **AppSheet UI**: 現状デッドコード（参照式なし）

## §4. 入所系 3 サービス追加時の課題

### 課題 1: 日次レコード設計と既存「同月最新日のみ反映」の矛盾

| 観点 | 既存 01相談記録 | 入所系 3 |
|---|---|---|
| データ粒度 | 「実績がある日数」分の行 | **暦日 1〜31 すべての行**（疎データ） |
| 加算の反映 | 「同月最新日のみ反映」で 1 行 | **「同月全件反映_カウント」必要**（日数集計） |
| GAS 実装 | 全行展開で OK | **暦日 LEFT JOIN 実装新規必要** |

### 課題 2: GAS の category 分岐拡張

ファイル: `001_レセプト生成_CloudSQL.gs`
- L469-472, L974-988 で `=== "障害児相談支援"` 分岐
- 入所系 3 用に新分岐追加必要

### 課題 3: 「反映条件」ロジックの実装責任

→ **採用方針**: CloudSQL クエリで吸収（GAS の SQL に GROUP BY / window function）

### 課題 4: GAS ファイル構成（ユーザー方針 2026-06-23）

**入所系の処理と改定分処理は新規 .gs にする / 既存相談分は残す**

```
gas/HopeCare_CloudSQL_移行版_ainotudoi/
├── 001_レセプト生成_CloudSQL.gs            [既存・温存]
├── 001_レセプト生成_入所系.gs              [新規]
├── 001_レセプト生成_6月改定_相談.gs        [新規・改定分]
├── 000_AppConfig.gs                       [既存]
├── 000_CloudSQL接続.gs                    [既存]
├── 000_callAppSheetApi.gs                 [既存]
└── 月次バックアップ_CloudSQL.gs            [既存]
```

## §5. AppSheet ↔ CloudSQL ↔ GAS のキー連携ポイント

| 連携点 | 詳細 |
|---|---|
| Form → CloudSQL | 加算名は **文字列で保存**（Row ID 参照ではない）。マスタリネーム禁止 |
| マスタ取得 | GAS は AppSheet DB を**直接読まない**。Bot が事前展開した結果を appDB スプシ経由で取得 |
| GAS → AppSheet | API v2 (ApplicationAccessKey) で **書き戻しのみ** |
| トリガ | AppSheet Bot の data change event のみ。Webhook / scheduled なし |
| テンプレ | リタリコ Excel テンプレを Drive コピー → A 列ラベル検索 → B 列 setValues |
| 数式保持 | 曜日列 (B) のみ `setFormulas`、他列は `setValues`。マージ時も B 列保護 |

## §6. 推奨設計（最終）

1. **加算マスタ** = HopeCareマスタ DB 共有のまま、入所系 3 サービス分の行を追加
2. **反映条件 Enum** = 既存「同月全件反映_カウント」を活用、日数加算は全てこの値で
3. **GAS** = 入所系処理を新規 `.gs` に分離。既存相談処理は温存
4. **CloudSQL SQL** = 暦日 LEFT JOIN を新規 SQL で吸収（窓関数 / generate_series）
5. **テンプレ** = リタリコ Excel から サンプル値を消した版を相談テンプレと同じ Drive フォルダに配置
