## §15. ⚠️ 既存（相談）vs 新規（入所系）の Excel 出力行構造の決定的な違い

> ユーザー追記事項（2026-06-22）

| 観点 | 既存（計画相談・障害児相談） | 新規（入所系3サービス） |
|---|---|---|
| 1 利用者ブロックの日次行数 | **実績日数分だけ行追加（可変）** | **暦日（1〜月末日）すべて固定で並ぶ** |
| 行の意味 | 各行 = 利用日 | 各行 = その月の暦日。データ部空 = 実績なし |
| ブロック行数 | 例: 5 日利用なら 5 行 | 例: 6 月なら 30 行（属性 + ヘッダ含めて 約 50 行） |
| GAS 出力時の振る舞い | CloudSQL 取得行をそのまま並べる | CloudSQL **疎データを暦日マスタと LEFT JOIN** して 28/29/30/31 行を生成 |
| 動的行番号への影響 | 実績件数で増分計算 | 月の日数（28/29/30/31）で増分計算 |

### GAS への追加要件（計画書に必ず反映）
1. **暦日生成関数**: 「サービス提供年月 → 1〜月末日 の配列」を返す関数を追加（うるう年対応）
2. **LEFT JOIN ステップ**: CloudSQL の実績データ × 暦日 → 暦日固定 28〜31 行 を生成
3. **空行の書込み制御**: B 列（曜日数式セル）は **触らない**、A 列（日番号）と C 列以降のデータ部 のみ書き込む
4. **動的行番号計算式の差別化**:
   - 既存: `nextBlockBase = currentBase + 属性行数 + 利用者の実績件数`
   - 新規（入所系）: `nextBlockBase = currentBase + 属性行数 + 月の暦日数(getDaysInMonth(year, month))`

### CloudSQL 設計はそのまま
- CloudSQL の 3 新テーブル（`child_care_entry_records` / `short_stay_records` / `daytime_temp_support_records`）は **疎データ** （実績日のみ INSERT）で確定。
- 暦日固定行は **Excel 出力ロジック側で生成**。DB 側に空行は持たない。
