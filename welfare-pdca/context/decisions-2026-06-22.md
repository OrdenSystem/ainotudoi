## 4. 「契約」概念のデータモデル（Q4 — DisabilityCard__c 拡張）

新規オブジェクト（Contract__c）は作らず、`DisabilityCard__c` に以下を **追加** する：

| 項目 API 名（仮） | 型 | リタリコ Excel 対応 | 用途 |
|---|---|---|---|
| `ContractRowNumber__c` | Number | 行 43「契約記入欄番号」 | 受給者証上の契約欄番号 |
| `ContractStartDate__c` | Date | 行 44「契約支給量開始日」 | 契約開始日 |
| `ContractEndDate__c` | Date | 行 45「契約支給量終了日」 | 契約終了日 |
| `MonthlyAllotmentDays__c` | Number | 行 46「日数」 | 月間支給量（日数） |
| `UpperLimitFacilityNumber__c` | **Text(10)** | 行 47「上限額管理事業所番号」 | 上限管理事業所番号 |
| `UpperLimitFacilityName__c` | **Text(80)** | 行 48「上限額管理事業所名」 | 上限管理事業所名 |
| `UpperLimitResult__c` | Picklist | 行 49「上限額管理結果」 | 上限管理結果（管理結果番号） |

> ⚠️ **2026-06-22 ユーザー指示**: 上限管理事業所は **Lookup(Office__c) を使わず、テキスト型 2 項目（番号 + 名前）のみ**で運用する。他プロジェクトでも同方針。理由: 他事業所が自社 Office__c マスタに存在しない可能性が高く、Lookup の参照整合性で運用が窮屈になるため。整合性は事業所番号の運用ルール（10 桁数字）で担保する。
| `ContractType__c` | Picklist | （契約 / 措置） | 児童入所施設で必須 |

実機ギャップ（Agent A 調査結果）と一致：
- DisabilityCard__c に **日数ベース支給量列が無い** → `MonthlyAllotmentDays__c` で解消
- DisabilityCard__c に **上限管理事業所参照が無い** → `UpperLimitFacility{Number/Name/__c}` で解消

> Cycle 1 で論点だった「契約オブジェクト新設 vs DisabilityCard 拡張」は **拡張に確定**。

## 6. ⚠️ 既存 GAS 請求 Excel 出力ロジックの厳守事項（ユーザー警告）

> 「請求 Excel 出力の GAS は以前すごく苦労した。あらゆる所に気を配って仕上げている。既存の GAS 処理を深く読むように計画に入れる」

以下の 4 点は **既存の挙動を完全に保持** すること。新サービス追加は「既存ロジックに 3 分岐追加」の形で行い、**既存ロジックの再設計はしない**：

### 6-1. 事業所情報セクションの加算項目は「請求アプリ」から取得
- 事業所全体で取る加算項目は **本体システム（HopeCareDX）とは別の請求アプリ**（`請求_HopeCareDX_愛の集い` / App ID `f6ddf60e-a346-4d4c-a143-eeb9aed81287`）から取得している
- → 入所系 3 サービス追加でも、**事業所単位の加算項目は請求アプリで管理**する
- → AppSheet 設計時に「請求アプリのどのテーブルにどう紐付くか」を spec 化必須

### 6-2. 市町村情報のユニーク数で以降の行番号が動的になる
- 市町村が複数ある事業所では市町村ブロックが繰り返され、それに応じて利用者ブロックの開始行がズレる
- **行番号をハードコードしている GAS 関数は、この動的計算ロジックを必ず通している**
- → 入所系追加でも **動的行番号計算ロジックを流用**、決して固定オフセットで決め打ちしない
- → 特に **日中一時支援（市町村事業）** は影響が大きい

### 6-3. テキスト置換処理が特殊
- 「日報Excel置換」シートで行うラベル→値の置換ロジックには **既存の特殊処理** がある（正確な内容は Agent B 解析待ち）
- → 既存の置換ルールに **新列・新ラベルのエントリを追加する形** で対応、既存エントリは変更しない

### 6-4. 数式で曜日セルが保持される
- 利用者欄の曜日セルは **数式で自動生成** されている（サービス提供年月入力で 1〜31 日の曜日が自動反映）
- GAS が値を書き込むと数式が壊れる → **書込対象範囲を厳密に限定** している
- → 入所系追加でも **既存の書込範囲制御ロジックを継承**、新ロジックで範囲を再定義しない

## 7. ⚠️ Excel 出力行構造の根本的な違い（既存 vs 新規）

> ユーザー追記事項（2026-06-22）

| 観点 | 既存（計画相談・障害児相談） | 新規（入所系3サービス） |
|---|---|---|
| 利用者ブロック内の日次行 | **実績がある日数分だけ行追加** | **暦日 1〜31 日（または月末日まで）すべての行が固定で並ぶ** |
| 空欄の意味 | 行自体が存在しない | 行は存在し、列の値が空 = 実績なし |
| 1 利用者ブロックの行数 | 可変（利用日数依存） | **固定（28〜31 行 + ヘッダ・属性領域）** |
| 月またぎ・空欄日の扱い | 行が無いだけ | 行はあり、データ部空 |

### CloudSQL 設計への影響
- **CloudSQL は疎データのまま**（実績がある日のみ INSERT）。これは決定。
- GAS が Excel 出力時に「暦日マスタ（1〜月末日）と LEFT JOIN」して、空行も含めて 28〜31 行を出力する。
- Excel 出力ロジックの **暦日生成 + LEFT JOIN ステップ** を新サービス分で追加必要。

### GAS の動的行番号計算への影響
既存 vs 新規で「利用者ブロック開始行 → 次の利用者ブロック開始行」の増分計算式が違う：

| | 既存 | 新規（入所系） |
|---|---|---|
| 増分式 | `currentBase + 属性行数 + 実績件数 + 余白` | `currentBase + 属性行数 + 暦日固定数(28〜31) + 余白` |
| ハードコード可否 | NG（実績件数で動的） | NG（月の日数で 28/29/30/31 と変動） |

→ category 別 + サービス提供月別の **暦日数取得関数** を GAS に追加する必要あり。

### 「市町村情報の動的行」との合成（decisions §6-2 と本節）

入所系の利用者ブロック開始行 = `事業所セクション末尾行 + 市町村ブロック数 × 5 + (前利用者ブロックの累計行数)`

このうち **「前利用者ブロックの累計行数」が暦日数 × N人** で動くため、市町村数と暦日数の**両方を動的計算**する必要がある。**ハードコード絶対禁止**。
